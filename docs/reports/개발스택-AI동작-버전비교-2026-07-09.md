## 4) AI가 실제로 동작하는 방식 (이해하기 쉽게)

### 4-1. 역할 분리

- `resume-api`: 사용자 요청/인증/도메인 저장 담당
- `resume-ai`: 생성·첨삭·탐지·공고분석의 AI 오케스트레이터
- `prompt-service`: 프롬프트 템플릿 버전 관리 + 렌더링
- `rag-service`: 임베딩/유사도 검색/컨텍스트 구성

### 4-2. 핵심: AI Gateway 오케스트레이션(서비스 호출 순서)

사용자가 궁금해한 포인트를 한 줄로 요약하면:

`resume-api`는 "업무 진입점", `resume-ai`는 "AI 워크플로우 오케스트레이터"입니다.

```mermaid
sequenceDiagram
    autonumber
    participant UI as resume-web / admin
    participant API as resume-api (Spring)
    participant AI as resume-ai (FastAPI Gateway)
    participant PROMPT as prompt-service
    participant RAG as rag-service
    participant LLM as OpenAI/LLM
    participant DB as PostgreSQL

    UI->>API: 자기소개서 생성/첨삭/탐지/공고분석 요청
    API->>DB: 사용자/공고/경험 데이터 조회
    API->>AI: AI 작업 요청 (내부 토큰 포함)

    alt 생성(cover letter generation)
        AI->>RAG: 경험+공고 기반 context 구성 요청
        RAG->>DB: 벡터 유사도 검색(pgvector)
        RAG-->>AI: context 패킷 반환
        AI->>PROMPT: 템플릿 렌더링 요청
        PROMPT->>DB: 활성 프롬프트 버전 조회
        PROMPT-->>AI: system/user prompt 반환
        AI->>LLM: 최종 프롬프트+컨텍스트로 생성 호출
        LLM-->>AI: 생성 결과
    else 첨삭/탐지/리뷰
        AI->>PROMPT: 작업별 프롬프트 렌더
        PROMPT-->>AI: 렌더된 프롬프트
        AI->>LLM: 분석/판정 호출
        LLM-->>AI: 결과 JSON
    else 공고분석(Job Analysis)
        AI->>PROMPT: JOB_ANALYSIS 프롬프트 렌더(v8)
        PROMPT-->>AI: 섹션 규칙 포함 프롬프트
        AI->>LLM: 추출 시도 (텍스트/비전)
        LLM-->>AI: 구조화 JSON 또는 실패
        AI->>AI: 실패시 rule-based fallback + postprocess
    end

    AI->>DB: ai_usage_logs 기록
    AI-->>API: 최종 결과 반환
    API-->>UI: 응답 반환
```

### 4-3. 대표 플로우: 공고 분석(Job Analysis)

```mermaid
flowchart TD
    IN[입력: TEXT / URL / PDF / IMAGE] --> NORM[정규화 normalize_text]
    NORM --> ROUTE{소스 타입}
    ROUTE -->|URL| FETCH[_fetch_url + html_to_text]
    ROUTE -->|PDF| PDF[_extract_pdf_base64]
    ROUTE -->|IMAGE| IMG[Vision 추출 시도]
    ROUTE -->|TEXT| TXT[텍스트 직접 분석]

    FETCH --> LLM
    PDF --> LLM
    TXT --> LLM
    IMG --> LLM

    LLM[LLM 추출 + JSON 파싱] --> OK{품질/파싱 성공?}
    OK -->|Yes| POST[postprocess_extraction]
    OK -->|No| RULE[rule-based 추출 fallback]
    RULE --> POST
    POST --> OUT[구조화 결과 저장/반환]
```

### 4-4. 핵심 포인트

- LLM 실패 시 **rule-based fallback**으로 결과를 비우지 않음
- 섹션 매핑 강제 (`지원 자격 -> required_skills`, `우대요건 -> preferred_skills` 등)
- 후처리에서 중복/오염 정리 후 최종 JSON 반환
- 프롬프트는 DB 버전(`V22`, Job Analysis `v8`)을 통해 운영

## 6) 실무 관점에서 보는 장단점

### 장점
- AI 기능을 서비스별로 분리해 장애 전파를 줄임
- prompt-service로 프롬프트 버전 관리가 가능해 실험/롤백이 쉬움
- rule fallback이 있어 API 키/모델 이슈 시에도 기본 기능 유지

### 주의점
- 프론트/파이썬 의존성이 범위형(`^`, `>=`)이라 환경별 미세 차이가 생길 수 있음
- AI 품질은 모델 자체보다도 프롬프트 버전/후처리 규칙 영향이 큼
- PWA/캐시(서비스워커)로 배포 직후 UI 확인 시 구버전이 보일 수 있음
